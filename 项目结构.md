# 项目结构

本文档说明当前仓库的目录划分、关键文件职责，以及主要运行链路，便于快速定位代码。

## 顶层目录

```text
Agent/
├─ app.py                  # Streamlit 聊天界面入口
├─ test.py                 # 命令行测试入口
├─ config/                 # YAML 配置
├─ agent/                  # Agent 组装、工具与中间件
├─ rag/                    # 向量检索与 RAG 服务
├─ model/                  # 大模型与向量模型工厂
├─ utils/                  # 配置、路径、日志、文件处理等通用工具
├─ prompts/                # 主提示词、RAG 提示词、报告提示词
├─ data/                   # 本地知识库文本与外部 CSV 数据
├─ logs/                   # 运行日志输出目录
├─ rag/chroma_db/          # 当前配置实际使用的 Chroma 持久化目录
├─ CLAUDE.md               # 提供给 Claude Code 的仓库工作说明
└─ 项目结构.md              # 本文档
```

## 目录说明

### 1. `app.py`

项目 Web 入口，使用 Streamlit 搭建聊天界面。

主要职责：
- 初始化 `ReactAgent`
- 维护 `st.session_state` 中的对话历史
- 通过流式方式展示模型输出

适合查看的场景：
- 想了解前端如何接入 Agent
- 想修改聊天 UI 或会话状态管理

---

### 2. `test.py`

命令行测试入口，用于不启动 Streamlit 时直接验证 Agent 响应。

主要职责：
- 接收命令行 query 参数
- 调用 `ReactAgent.execute_stream()`
- 在终端打印流式输出

适合查看的场景：
- 快速 smoke test
- 调试 Agent 输出，不想经过 UI

---

### 3. `config/`

集中管理 YAML 配置。

当前关键文件：
- `config/agent.yml`：Agent 相关配置，例如外部数据路径、模型名、API Key
- `config/rag.yml`：RAG 模型相关默认配置
- `config/chroma.yml`：向量库参数，如持久化目录、检索条数、分块大小
- `config/prompts.yml`：提示词文件路径映射

说明：
- 这些配置会在模块导入时被读取并缓存
- 修改后通常需要重启 Python 进程才能生效

---

### 4. `agent/`

Agent 核心逻辑所在目录。

#### `agent/react_agent.py`
负责组装 LangChain Agent：
- 指定聊天模型
- 加载系统提示词
- 注册 tools
- 注册 middleware

它是整个智能客服运行链路的核心入口。

#### `agent/tools/agent_tools.py`
定义提供给 Agent 的工具，包括：
- `rag_summarize`：走本地知识库检索总结
- `get_weather`：模拟天气信息
- `get_user_location`：模拟用户位置
- `get_user_id`：模拟用户 ID
- `get_current_month`：模拟当前月份
- `fetch_external_data`：从 CSV 获取用户使用记录
- `fill_context_for_report`：为报告场景切换提示词上下文

说明：
- 多个工具是 demo/stub 实现，不是真实外部服务
- `fetch_external_data` 会把 CSV 数据缓存到进程内全局字典中

#### `agent/tools/middleware.py`
中间件负责两件关键事情：
- 记录工具调用与模型调用日志
- 根据上下文动态切换提示词

最关键的机制是：
- 当 `fill_context_for_report()` 被调用时
- 中间件会把 `runtime.context["report"]` 置为 `True`
- 后续模型调用会从主提示词切到报告提示词

这就是“普通问答”和“报告生成”共用一套 Agent，但能切换行为的原因。

---

### 5. `rag/`

负责知识库检索与摘要生成。

#### `rag/rag_service.py`
封装 `RagSummarizeService`，完整流程为：
1. 从向量库检索相关文档
2. 将检索结果拼接成上下文字符串
3. 套用 RAG 提示词模板
4. 调用模型生成总结回答

#### `rag/vector_store.py`
负责向量库的加载、切分、入库与检索。

主要职责：
- 使用 Chroma 持久化向量数据
- 从 `data/` 读取 `.pdf` 和 `.txt` 文件
- 使用 `RecursiveCharacterTextSplitter` 切块
- 通过 `md5.text` 去重，避免重复入库
- 暴露 retriever 给 RAG 服务使用

注意：
- 当前文件扫描是**非递归**的，只会读取 `data/` 目录下的直接子文件
- 当前配置实际使用的向量库存储目录是 `rag/chroma_db`

---

### 6. `model/`

封装聊天模型和向量模型的创建逻辑。

#### `model/factor.py`
主要职责：
- 创建 `ChatTongyi` 聊天模型实例
- 创建 `DashScopeEmbeddings` 向量模型实例
- 通过工厂模式统一初始化方式

说明：
- 模型实例在模块导入时就会创建
- 聊天模型、向量模型都依赖配置文件中的模型名和 API Key

---

### 7. `utils/`

项目公共工具目录。

关键文件：
- `utils/config_handler.py`：读取 YAML 配置并暴露模块级全局配置对象
- `utils/path_tool.py`：将相对路径统一转换为项目根目录下的绝对路径
- `utils/prompt_loader.py`：读取 prompt 文件内容
- `utils/file_handler.py`：处理 PDF/TXT 加载、目录遍历、MD5 计算
- `utils/logger_handler.py`：统一日志初始化，输出到 `logs/`

这一层是各模块的基础依赖，尤其是路径与配置工具被频繁使用。

---

### 8. `prompts/`

存放 Agent 行为定义文本。

关键文件：
- `prompts/main_prompt.txt`：主客服提示词，约束普通问答和工具调用流程
- `prompts/rag_summarize.txt`：RAG 总结提示词，要求回答严格基于检索内容
- `prompts/report_prompt.txt`：报告生成场景提示词

说明：
- 该项目的很多业务约束不是写在 Python 里，而是写在 prompt 里
- 报告工具调用顺序、RAG 输出边界等都依赖这里的文本约束

---

### 9. `data/`

本地数据目录。

主要内容分两类：

#### 知识库资料
例如：
- `维护保养.txt`
- `故障排除.txt`
- `选购指南.txt`
- 若干 PDF / TXT 问答资料

这些文件会被导入 Chroma，作为 `rag_summarize` 的检索来源。

#### 外部结构化数据
- `data/external/records.csv`

用于模拟用户每月使用记录，供报告生成工具读取。

---

### 10. `logs/`

运行日志目录。

日志由 `utils/logger_handler.py` 统一生成，按日期命名。适合排查：
- 工具调用是否执行
- 模型调用前的消息情况
- 文档加载和向量库构建过程中的异常

## 主要运行链路

### 普通问答链路

```text
用户输入
  ↓
app.py / test.py
  ↓
ReactAgent
  ↓
主提示词 + tools + middleware
  ↓
如有需要调用 rag_summarize / 其他工具
  ↓
返回最终回答
```

### 报告生成链路

```text
用户请求“生成/查询使用报告”
  ↓
Agent 调用 fill_context_for_report
  ↓
middleware 将 report 上下文标记为 True
  ↓
动态切换到 report_prompt.txt
  ↓
Agent 获取 user_id / month / external data
  ↓
生成中文 Markdown 报告
```

### 知识库构建链路

```text
data/ 下 PDF/TXT 文件
  ↓
utils.file_handler 加载文件
  ↓
vector_store.py 分块
  ↓
embedding 模型向量化
  ↓
写入 Chroma 持久化目录
  ↓
rag_service.py 检索使用
```

## 快速定位建议

- 想改聊天入口：看 `app.py`
- 想改 Agent 装配：看 `agent/react_agent.py`
- 想增删工具：看 `agent/tools/agent_tools.py`
- 想改报告模式切换：看 `agent/tools/middleware.py`
- 想改检索逻辑：看 `rag/rag_service.py` 与 `rag/vector_store.py`
- 想改模型或配置读取：看 `model/factor.py` 与 `utils/config_handler.py`
- 想改提示词：看 `prompts/` 和 `config/prompts.yml`

## 当前仓库的几个特点

- 更偏 Demo / 课程式结构，很多外部能力是模拟实现
- 业务行为有一部分写在 prompt 中，而不是 Python 逻辑里
- 配置、模型、RAG 服务存在较多“导入即初始化”的模式
- 仓库里存在多个 `chroma_db` 目录，但当前配置实际指向 `rag/chroma_db`
