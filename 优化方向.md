# 后端优化方向

## 目标

当前后端已经具备最小可用的问答能力，但还存在**启动脆弱、接口能力偏弱、RAG 强依赖初始化、联调体验一般**等问题。优化目标是把后端从"能跑的 Demo"提升到"能稳定给前端提供服务"的状态。

---

## 一、当前主要问题（已解决部分）

### 1. 服务启动阶段耦合过重 ✅ 已解决
改为延迟初始化：模型、Agent、RAG 均在使用时才创建，`/health` 不依赖模型。

### 2. RAG 导入阶段初始化 ✅ 已解决
改为 `get_rag_service()` 懒加载，失败时返回"知识库暂不可用"而不是让接口 500。

### 3. API 能力扩展 ✅ 已解决
- 请求体增加 `history`、`session_id`、`user_context`
- 新增 SSE 流式接口 `/api/chat/stream`
- 统一响应结构 `success/message/data` + `trace_id`

### 4. 异常处理统一 ✅ 已解决
全局异常处理器，区分 400/502/503/500。

### 5. 工具层去随机 ✅ 已解决
- `get_user_location()` → 优先前端传入 → IP 自动定位 → 兜底
- `get_user_id()` / `get_current_month()` → 优先前端传入 → 确定性默认值
- `get_weather()` → wttr.in 真实天气，失败兜底 Demo

### 6. API Key 独立配置 ✅ 已解决
`api.yml` 独立文件，加入 `.gitignore`。

---

## 二、知识库（RAG）专项优化

以下是当前 RAG 链路中存在的具体问题及优化方向。

---

### 1. 检索策略过于基础，缺乏质量保障
文件：[end/rag/vector_store.py](end/rag/vector_store.py)、[end/rag/rag_service.py](end/rag/rag_service.py)

**当前状态：**
- 仅使用 Chroma 默认的相似度检索，固定 `k=3`
- 没有任何相关性阈值过滤 —— 即使相似度为 0 的文档也会被返回
- 没有重排序（re-ranking），完全依赖 embedding 相似度排序
- LLM 收到的 3 条参考资料可能和问题关系不大

**影响：**
- 检索到的文档可能不相关，导致 RAG 回答质量不稳定
- 用户问"耗材怎么更换"，可能被返回"产品规格参数"而不是"更换步骤"
- 无法区分"找到了高度相关的资料"和"勉强找到一点沾边的内容"

**优化方向：**
- 增加相似度阈值过滤，低于阈值的文档直接丢弃
- 引入重排序（如 `bge-reranker`），在粗排后再做一次精排
- 支持调整 `k` 值 —— 简单问题用 k=2，复杂问题用 k=5
- 在检索日志中记录：匹配文档数、最高相似度分数

---

### 2. 检索与生成耦合过重，无法复用好
文件：[end/rag/rag_service.py#L31-L52](end/rag/rag_service.py#L31-L52)

**当前状态：**
- `rag_summarize()` 一个方法做了三件事：检索 → 拼上下文 → 调 LLM 生成
- 检索结果不能单独获取，无法只展示"相关文档"
- 上下文拼接逻辑是内联的字符串拼接，格式固定

**影响：**
- 如果想做"只看参考资料不生成回答"的功能，没法复用
- 如果想把检索到的文档来源返回给前端展示，当前结构不支持
- Prompt 模板、检索、生成三者绑死在一起

**优化方向：**
- 拆成三个独立方法：
  - `search(query)` → 返回检索结果（文档列表 + 分数）
  - `format_context(docs)` → 将文档格式化为 Prompt 上下文
  - `generate(context, query)` → 基于上下文生成回答
- 对外暴露检索接口（如 `/api/rag/search`），让前端也能直接用检索结果
- 上下文格式化支持不同策略（简洁版 / 详细版 / 带来源引用版）

---

### 3. 知识库构建是一次性的，无法增量更新
文件：[end/rag/vector_store.py#L31-L120](end/rag/vector_store.py#L31-L120)

**当前状态：**
- `load_document()` 基于 MD5 去重 —— 同一文件只能加一次
- 没有删除文档的能力
- 文件内容改了但因为 MD5 变了，会被当成新文档加进去，旧向量仍然存在

**影响：**
- 知识库内容更新后，旧数据和新数据同时存在，LLM 可能看到矛盾信息
- 删除过期文档需要手动清空 Chroma 目录重建
- 没有增量更新能力，每次更新都要全量重建

**优化方向：**
- 增加文档删除能力：按文件名或文档 ID 从 Chroma 中删除
- 支持更新文档：检测到同名文件内容变化时，先删旧的再加新的
- MD5 去重改为"文件路径 + MD5"组合键，同一路径的文件更新视为替换
- 增加构建状态 API，让前端/运维知道知识库有多少文档、最后更新时间

---

### 4. Chunk（分块）策略固定，不适应异构内容
文件：[end/rag/vector_store.py#L20-L25](end/rag/vector_store.py#L20-L25)、[end/config/chroma.yml](end/config/chroma.yml)

**当前状态：**
- `chunk_size=200`、`chunk_overlap=20`，对所有文件一视同仁
- 分隔符是通用中文标点：`["\n","\n\n","。","？","！","；",".","!","?",";"]`
- PDF 和 TXT 用同一个分块配置

**影响：**
- 200 字的分块对技术文档可能过小，对一个段落都没切完
- 固定重叠可能导致关键信息被切在两个 chunk 之间，检索时丢失上下文
- 没有按文档类型（FAQ / 说明书 / 规格表）做差异化处理

**优化方向：**
- 按文档类型或目录区分 chunk 策略：FAQ 用小 chunk，说明书用大 chunk
- 引入语义分块（如按段落、按标题层级），而不是纯字符数切割
- 在配置中支持多套 chunk 策略：`chunk_strategies: { faq: {...}, manual: {...} }`
- 分块时保留元数据（来源文件、页码、章节标题）

---

### 5. 检索结果缺少来源引用，用户无法验证
文件：[end/rag/rag_service.py#L45-L52](end/rag/rag_service.py#L45-L52)

**当前状态：**
- RAG 返回的只是 LLM 生成的纯文本回答
- 没有附带引用来源（如"来源于《产品手册》第 3 页"）
- Prompt `rag_summarize.txt` 也没有要求模型输出来源

**影响：**
- 用户看到回答后无法判断信息是否可靠
- 如果 LLM 出现幻觉，用户无法交叉验证
- 前端无法展示"参考来源"卡片

**优化方向：**
- 在 Prompt 中要求模型在回答末尾附带来源引用
- 上下文中携带 `doc.metadata`（文件名、页码），让模型能引用
- 在 API 响应 `data` 中增加 `sources` 字段返回原始文档信息
- 前端展示"参考来源"可点击链接

---

### 6. 缺少检索缓存，重复查询浪费资源
文件：[end/rag/rag_service.py](end/rag/rag_service.py)

**当前状态：**
- 每次 `rag_summarize()` 调用都完整走一遍：embedding 检索 → LLM 生成
- 用户连续问相似问题时，每次都重新检索 + 重新生成
- 没有语义缓存（semantic cache）或简单的结果缓存

**影响：**
- embedding API 调用次数 = 用户提问次数，成本线性增长
- 多轮对话中用户追问时，每次都要重新检索

**优化方向：**
- 对检索结果做短期内存缓存（同一 session 内相同 query 不重复检索）
- 引入语义缓存：相似度 > 0.95 的问题直接返回缓存结果
- 将高频问题的检索结果预热到内存中

---


### 8. 缺少知识库健康状态的可观测性
文件：[end/rag/vector_store.py](end/rag/vector_store.py)、[end/rag/rag_service.py](end/rag/rag_service.py)

**当前状态：**
- 没有接口可以查看知识库状态（文档数、最后更新时间、向量维度等）
- 检索失败时只有日志，对外不透明
- 无法区分"知识库没建"、"检索无结果"、"embedding 不可用"三种情况

**影响：**
- 运维不知道知识库是否正常工作
- 前端/用户看到"未检索到参考资料"时无法判断原因
- 排查 RAG 问题需要登录服务器看日志

**优化方向：**
- 增加 `/api/rag/status` 接口，返回：
  - 知识库文档总数
  - 最后构建时间
  - embedding 模型状态
  - 最近检索成功率
- 检索失败时返回结构化原因：`{ reason: "no_index" | "no_match" | "embedding_error" }`
- 增加检索耗时和命中率的监控日志

---

### 9. RAG Prompt 可配置性不足
文件：[end/prompts/rag_summarize.txt](end/prompts/rag_summarize.txt)、[end/rag/rag_service.py](end/rag/rag_service.py)

**当前状态：**
- RAG Prompt 是写死的文本文件
- 上下文格式是硬编码的字符串拼接：`"参考资料{counter}：参考资料内容{doc.page_content}|参考元数据{doc.metadata}\n"`
- 无法根据不同场景切换 Prompt

**优化方向：**
- 上下文格式模板化，从配置文件读取
- 支持多套 RAG Prompt：详细回答 / 简洁回答 / 仅列举事实
- 由前端或工具层选择使用哪套 Prompt

---

## 三、知识库优化优先级

### P0：先保证检索质量可控
1. 增加相似度阈值过滤
2. 检索日志记录匹配分数
3. 区分"无结果 / 无知识库 / 检索失败"

### P1：再提升可维护性
1. 检索与生成解耦
2. 增加来源引用
3. 支持递归子目录加载

### P2：最后做进阶能力
1. 语义缓存
2. 重排序
3. 多知识库 / 按类型分 chunk 策略
4. 知识库状态 API
